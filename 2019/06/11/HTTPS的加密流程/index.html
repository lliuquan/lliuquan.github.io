title: Https的加密流程
date: 2019-05-25 15:57:35
categories: “计算机网络” #分类
tags: #标签

​ - Https
​ - 加密

本文主要讲解了Https的加密流程，如何生成密钥，以及对称性加密与非对称性加密的区别等

一. 概述

​ 早期的计算机属于单机状态，保证数据的安全性主要依靠加密算法的可靠性，即使拿到了数据密文，来恢复明文也是十分地困难的，之前使用的加密算法一般是DES对称性加密算法，现在使用较少，因为现代的计算能力，24小时内即可破除。

​ 后来发展到了网络时代，数据本身在传输层不会做加密处理，为了确保数据的安全性，人们开始使用了SSL协议。SSL协议工作在传输层与应用层之间，HTTPS的全称是HTTP over SSL ，就是在HTTP传输上增加了一层SSL协议的加密功能。

​ 我们第一次想到的其实还是加密算法，在传输的过程中先加密，但是加密之前得先生成密钥，但是密钥又靠什么来进行安全地传输呢？再次加密？这就变成了递归的问题。

​ 新的算法就此出现了——RSA算法，一种非对称性的算法

二.HTTP传输的问题

  1. 信息篡改——修改通信的内容
  2. 信息劫持——拦截到信息通信的内容
  3. 一般不会采用同一个秘钥，而是采用不同的秘钥加解密

三. 非对称性加密

​ 将密码革命性地分成了公钥和私钥，因为密钥不相同，所以称之为非对称性加密，而它的安全性是在于大质数分解的困难性。

​ 它的缺点是加密和解密的耗时长，只适合对少量数据进行处理，但是如果使用非对称性加密对密钥进行传输，就能解决之前所存在的问题。

​ 私钥加密后的密文，只要是公钥，都可以解密，但是反过来公钥加密后的密文，只有私钥可以解密。私钥只有一个人有，而公钥可以发给所有的人。

​ 简要的传输流程：在实际传输过程中，把公钥发送给对方，一方发送信息的时候，使用另一方的公钥进行加密生成密文，收到密文的一方在用公钥进行解密。（后面有补充）

公钥与私钥特点：

  • 公钥是开放给所有人的，但私钥是需要保密的，存在于服务端
  • 服务器端 server 向 client 端（A、B…..）的信息传输是不安全的：因为所有人都可以获取公钥
  • 但 client 端（A、B…..）向 server 端的信息传输确实安全的：因为私钥只有 server 端存在

四. 建立非对称性加密传输的流程

  1. 甲告诉乙，使用RSA算法来进行加密，乙说：好。
  2. 甲和乙分别根据RSA生成一对密钥，互相发送公钥。
  3. 甲使用乙的公钥来给乙加密报文信息。
  4. 乙收到信息，并用自己的密钥进行解密。
  5. 乙按照相同的方式给甲发送信息，甲使用相同的方式进行解密。

​ 信息通信采用http是不安全的，存在信息劫持、篡改的风险，https是加密传输，是安全的通信。

​ 对于https加密的过程，我们首先介绍的对称加密，采用对称加密进行通信存在秘钥协商过程的不安全性，因此我们采用了非对称加密算法解决了对协商过程的加密，因此https是集对称加密和非对称加密为一体的加密过程。

五. 当中存在的重要问题：

公钥被替换的情况

在发送公钥的途中，黑客去截取了甲的信息，并且严刑拷打之后知道了使者是去送公钥的，隐藏起甲发送给乙的公钥，然后自己生成一对密钥，冒充甲，然后交换公钥。这就形成了信任危机——我不知道这位送信使者是真的还是假的

信任危机的解决方法——CA/证书机构

​ 为了确保送信使者的真假身份，引入了一个权威的机构——CA，它是负责管理和签发证书的第三方机构，作用是检查证书持有者身份的合法性，并签发证书，以防证书被伪造或篡改.

​ CA 证书就是CA颁发的证书。 CA证书也就我们常说的数字证书，包含

  • 证书拥有者的身份信息
  • CA机构的签名
  • 公钥和私钥

​ 身份信息用于证明证书持有者的身份；CA签名用于保证身份的真实性；公钥和私钥用于通信过程中加解密，从而保证通讯信息的安全性。

TLS/SSL的区别

​ TLS可以理解为SSL协议的3.0版本的升级，TLS1.0版本也会标识为SSL3.1版本。但是对于大的协议栈SSL和TLS没有大的区别。

SSL证书验证过程

​ 服务器发送了一个SSL证书给客户端，SSL 证书中包含的具体内容有证书的颁发机构、有效期、公钥、证书持有者、签名，通过第三方的校验保证了身份的合法，解决了公钥获取的安全性

（1）首先浏览器读取证书中的证书所有者、有效期等信息进行一一校验

（2）操作系统内部校验证书是否是合法的机构颁发，浏览器开始查找操作系统中已内置的受信任的证书发布机构CA，与服务器发来的证书中的颁发者CA比对，用于校验证书是否为合法机构颁发

（3）如果找不到，浏览器就会报错，说明服务器发来的证书是不可信任的。

（4）如果找到，那么浏览器就会从操作系统中取出颁发者 CA 的公钥，然后对服务器发来的证书里面的签名进行解密

（5）浏览器使用相同的hash算法计算出服务器发来的证书的hash值，将这个计算的hash值与证书中签名做对比

（6）对比结果一致，则证明服务器发来的证书合法，没有被冒充

（7）此时浏览器就可以读取证书中的公钥，用于后续加密了

六. HTTPS的握手流程

​

第一次发生 Client Hello 协议的请求：在其中最重要的信息是 Cipher Suites 字段，客户端会告诉服务器端自己支持哪些加密的套件。

​ 第二，服务器端收到客户端发来的Client Hello 请求后，返回一系列的协议的数据，并以一个没有数据内容的Server Hello Done 作为结束。
